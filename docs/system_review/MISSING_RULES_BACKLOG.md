# MISSING RULES BACKLOG

| Priority | Missing Rule / Issue | Module | Affected Role | Risk | Recommended Fix |
|---|---|---|---|---|---|
| P0 Critical | Direct order CRUD endpoint can bypass workflow services, notifications, provider checks, and status logging | Orders | Admin | Unsafe state changes and deletions | Remove update/delete for normal admins or route all writes through workflow services |
| P0 Critical | Generic status endpoint can move orders to `READY_FOR_DELIVERY` or `COMPLETED` without final-document validation | Orders / Workflow | Employee, Admin | Orders can close without required artifacts | Block these statuses in generic status API and force dedicated upload/complete services |
| P0 Critical | Uploading the first missing document clears all requested missing-document requirements and reopens review | Orders / Documents | Client, Employee | Orders can resume with incomplete customer response | Recompute remaining missing types and reopen only when all required missing docs are supplied |
| P0 Critical | Provider order-detail API exposes customer PII, full note payload, and all document metadata beyond least privilege | Orders / Documents | Provider | Data leakage | Create provider-safe serializers for order detail, notes, and documents |
| P1 High | Provider approval exists in model but is missing from normal admin UI/API workflow | Providers | Admin, Employee, Provider | Providers can be created but remain unassignable | Add approval fields and audit trail to provider management screen/API |
| P1 High | Admin-created users/providers can receive predictable default passwords | Accounts / Providers | Admin, Provider, Employee, Client | Account compromise risk | Require explicit password set or invitation/reset flow |
| P1 High | Manual notification permission exists for employees, but no employee UI exists | Notifications | Employee | Operational gap and off-system communication | Add employee notification composer with templates and audit |
| P1 High | Notification templates exist in the schema but are not used by runtime send logic | Notifications | Admin, Client, Employee, Provider | Template maintenance has no effect | Resolve templates at send time and record template usage on notification rows |
| P1 High | Document verification decisions are not audited | Documents / Audit | Employee, Admin, Provider, Client | No trace for approval/rejection of sensitive files | Add audit entries for verify/reject actions with note and actor |
| P1 High | Admin user management can create `provider` users without `ProviderProfile` | Accounts / Providers | Admin, Provider | Broken provider accounts | Prevent provider role creation in generic user screen or auto-create the profile |
| P1 High | Admin/provider/document/order custom CRUD endpoints are not consistently audited | Accounts / Orders / Documents / Providers / Services | Admin | Low accountability for sensitive changes | Add service-layer audit logging for all write endpoints |
| P2 Medium | Customer upload endpoint accepts arbitrary `document_type` values after order creation | Documents | Client, Employee | Dirty data and unclear requirements | Restrict upload types to service requirements plus a controlled `other` path |
| P2 Medium | Employees can access report APIs but have no route or page | Reports | Employee | Hidden capability and inconsistent operations | Add read-only employee reports dashboard or remove permission |
| P2 Medium | Notification center shows employees/providers notifications tied to accessible orders even when they are not recipients | Notifications | Employee, Provider | Potential message leakage | Separate `recipient` feed from `order-related` feed or redact message content |
| P2 Medium | No frontend screen for service required documents or service-provider assignments | Services | Admin | Admin must rely on Django admin / raw backend | Add dedicated admin screens with simple controls |
| P2 Medium | No frontend screen for payments even though backend supports customer/admin payment flows | Payment | Client, Admin | Workflow blind spot | Add payment list/status screens or remove unused backend until needed |
| P2 Medium | `SystemSetting` is free-form JSON with no schema or key governance | Accounts | Admin | Misconfiguration risk | Replace free-text key/value with curated setting registry and typed controls |
| P3 Low | Current DB snapshot has no configured `SystemSetting` rows | Accounts | Admin | Admin UI exists but nothing is configured | Seed baseline settings with clear descriptions |
| P3 Low | Current DB snapshot has no configured notification templates | Notifications | Admin | Template admin is empty | Seed core templates or hide template screen until runtime uses them |
| P3 Low | Support role exists in backend/frontend guards but has no dedicated explanation or distinct UX | Accounts / Frontend | Employee, Admin | Role confusion | Document support behavior or merge it formally into employee role |
